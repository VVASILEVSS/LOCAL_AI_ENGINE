### Архитектура MZZ4 (твоя настройка)

4 независимых ZigZag слоя, каждый на своём TF, разный цвет:

| Слой | TF | Depth | Deviation | Backstep | MaxBar | Цвет |
|------|-----|-------|-----------|----------|--------|------|
| 1 | M15 | 12 | 5 | 3 | 5000 | Aqua |
| 2 | H1 | 12 | 5 | 3 | 1000 | Red |
| 3 | H4 | 12 | 5 | 3 | 350 | Yellow |
| 4 | D1 | 12 | 5 | 3 | 150 | Magenta |

Параметры одинаковые (12,5,3), но **lookback разный**: M15=5000 свечей, D1=150. Альтернативные настройки (закомментированные): `Depth=14,28,102 / Dev=1,8,21 / Backstep=3,5,12` — параметры растут для старших TF.

**Ключевой принцип MZZ4:** каждый TF считает ZigZag НА СВОИХ свечах через `iLow/iHigh(NULL, GrossPeriod_[x], ...)`. Результат проектируется на текущий график через `ZigZagDT` — находит реальный bar внутри GrossTF свечи. У каждого TF — **свои собственные пивоты → свои zone boundaries**.

## 2. Корневая причина контаминации зон (BTC) и сужения (ETH)

### Симптомы

**BTC 14:24 (правильные зоны):**
- D1: [57758.60 - 97932.10] | BOS↓ 62232.10 age=22
- H4: [61806 - 66809.61] | BOS↑ 64497.40 age=6
- H1: [64456.70 - 65260] | BOS↑ 64950 age=11
- M15: [64456.70 - 65121.84]

**BTC 14:44 (сломанные — контаминация):**
- D1: [57758.60 - 97932.10] ✅ (same)
- H4: **[57758.60 - 67255.40]** ❌ lower = D1 low
- H1: **[57758.60 - 65260]** ❌ lower = D1 low
- M15: **[62634.96 - 65121.84]** ❌ lower упал
- Confluence: `57758.60 | TF: ['1D','4H','1H','15M']` — D1 low протёк во все TF

**ETH 14:26 (зужены):**
- D1: [1503.60 - 3403.80] | BOS↑ 1779 age=12
- H4: [1748 - 1896.90] ✅ (correct)
- H1: [1863.30 - 1896.90] ❌ сужен (реальный low ~1779)
- M15: [1863.30 - 1893.88] ❌ сужен

### Root cause — RAW LLM output

Поймал RAW output GLM для ETH:

```json
"tf_zones": {
    "15m": { "range": [1503.6, 1896.9] },   ← lower = D1 lower!
    "1h":  { "range": [1503.6, 1896.9] },   ← lower = D1 lower!
    "4h":  { "range": [1503.6, 1896.9] },   ← lower = D1 lower!
    "1D":  { "range": [1503.6, 3403.8] }
}
```

**LLM копирует D1 lower (1503.6) во все TF.** Потом POST-LLM фильтр видит дубликаты → confluence фильтр → VP fallback → узкие зоны.

### Цепочка бага

```
ZigZag benchmark (correct per-TF)     ← structure.py даёт правильные зоны
    ↓
LLM context (correct zones в prompt)  ← LLM видит правильные зоны
    ↓
LLM output (D1 lower во всех TF) ❌   ← GLM гомогенизирует lower
    ↓
POST-LLM confluence filter            ← ловит дубликаты
    ↓
VP fallback (narrow)                  ← заменяет на узкие зоны
    ↓
TG message (суженые/контаминированы)  ← пользователь видит баг
```

### ZigZag benchmark vs LLM output (ETH)

| TF | ZigZag (correct) | RAW LLM | TG (после фильтра) |
|----|------------------|---------|-------------------|
| H4 | [1750 - 1896] ✅ | [1503.6 - 1896.9] ❌ | [1748 - 1896.9] ✅ |
| H1 | [1779 - 1848] ✅ | [1503.6 - 1896.9] ❌ | [1863.3 - 1896.9] ❌ сужен |
| M15 | [1751 - 1896] ✅ | [1503.6 - 1896.9] ❌ | [1863.3 - 1893.88] ❌ сужен |

**Вывод:** твой `structure.py` уже работает как MZZ4 — даёт правильные per-TF зоны. Баг не в ZigZag, баг в LLM.

## 3. Варианты фикса — прошу подумать

### Вариант A — ZigZag fallback вместо VP
Когда POST-LLM отклоняет зону (confluence/min_span), брать зону из ZigZag benchmark (структурную), а не из Volume Profile. ZigZag = структура, VP = объём. Для SMC структурные зоны важнее.

### Вариант B — Валидация LLM output
Если LLM зона для TF совпадает с D1 зоной (lower одинаковый) → это гомогенизация → заменить на ZigZag зону для этого TF. Точечный детектор контаминации.

### Вариант C — Confluence фильтр + BOS check (уже частично реализован)
Как мой патч C для min_span (web_dashboard.py:397-404), добавить BOS check в confluence фильтр: если зона имеет валидный BOS — не заменять даже если совпадает с parent. H1 имеет BOS↑ 1888 age=11, но confluence фильтр всё равно заменил → патч C не спас.

### Вариант D — Промпт
Добавить в PRO_TA_USER_PROMPT явное правило: "Используй zone boundaries из zigzag context для каждого TF. НЕ копируй boundaries из старших TF в младшие."

### Вариант E — ZigZag authoritative + LLM narrative only (радикальный)
1. Zone boundaries берутся из ZigZag benchmark (per-TF, как в MZZ4)
2. LLM заполняет только: trend, structure, signal, BOS interpretation, wave phase, accumulation, SL/TP
3. LLM не может переопределять `tf_zones.range` — только читать из контекста
4. POST-LLM фильтр убирается (не нужен — зоны уже корректные из ZigZag)
5. BOS берётся из `structure.py` (уже есть в benchmark)

Убирает и контаминацию, и confluence фильтр, и VP fallback. LLM становится интерпретатором, не генератором зон.
