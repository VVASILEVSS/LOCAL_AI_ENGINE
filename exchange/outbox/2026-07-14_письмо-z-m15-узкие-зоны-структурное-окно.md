## Почему `_enforce_zone_uniqueness` не фиксит 15m

```python
tf_expand = [("1D", 0.025), ("4H", 0.015), ("1H", 0.0075), ("15M", 0.0), ("5M", 0.0)]
```

- expand_pct для 15m = 0.0 — **нет расширения**
- expand_pct для 5M = 0.0 — **нет расширения**
- Расширение работает только для parent: D1 → H4 → H1
- 15m = child, остаётся как есть

Плюс: fallback chain в `_fill_missing_tf_zones` не вызывается — LLM вернул зону → `continue` → VP/ZigZag fallback не используется.

---

## Корневая причина

LLM "видит" на 15m графике последние 5-10 свечей в сжатии и ставит зону = этот микроканал. Модель **не видит структурный range** — законченную предыдущую структуру и текущую после слома. ZigZag даёт правильный range (1748-1848), но LLM его игнорирует, потому что:

1. ZigZag контекст в промпте — текстовые числа, не визуальная структура
2. На самом 15m графике последние свечи в узком канале → LLM копирует это сжатие
3. Нет указания в промпте, что зона = структурный range, а не текущий микроканал

---

## Решение: структурное окно ТФ

Главная идея: **нужно чтобы модель видела окно ТФ с двумя структурными движениями** — законченную предыдущую структуру и после слома текущую. Тогда уровни будет видеть.

Сейчас на 15m графике LLM видит последние ~20-30 свечей в боковике и ставит зону = этот микроканал. Но если показать ему окно где видно:
- **Предыдущая структура** (например, нисходящий тренд 1748 → 1793)
- **Слом** (пробой 1793 вверх)
- **Текущая структура** (восходящая от 1793)

...то модель увидит реальные структурные уровни: 1748 (low предыдущей структуры) и 1793 (уровень слома). Это даст правильную зону.

### Как это реализовать

**Вариант 1: Ограничить окно ZigZag** — вместо 100 свечей показать ZigZag за последние 40-50 свечей, где видно 2 структурных движения (предыдущая + текущая). Сейчас ZigZag берёт 200 свечей, показывает огромный range (1748-3403 на 1D), и LLM не понимает какой уровень актуальный.

**Вариант 2: Структурные маркеры на графике** — на 15m графике отметить вертикальной линией "слом структуры" и подписать "до слома: нисходящий тренд, после слома: боковик". LLM увидит две структуры и поставит зону = range актуальной структуры.

**Вариант 3: В промпте явно указать** — "Зона для каждого ТФ = range от последнего слома структуры до текущего момента. НЕ ставь зону = последние 5 свечей в сжатии."

## Варианты фикса (технические)

### A) Min-span validation (post-hoc)

В `enforce_risk_rules`, после `_enforce_zone_uniqueness`:

```python
# Если LLM зона по любому ТФ У́ЖЕ чем VP value area → удалить → fallback
for tf_key, zone in tf_zones.items():
    if not isinstance(zone, dict):
        continue
    span = abs(zone.get("upper", 0) - zone.get("lower", 0))
    vp_zone = vp_data.get(tf_key)  # VAH/VAL
    if vp_zone and span < (vp_zone["VAH"] - vp_zone["VAL"]) * 0.5:
        del tf_zones[tf_key]  # _fill_missing_tf_zones подставит VP
```

Плюсы: универсально, использует реальные VP данные, source tracking уже есть.
Минусы: VP может быть тоже узким (если сжатие реальное).

### B) Расширить `_enforce_zone_uniqueness` на 15m

```python
tf_expand = [("1D", 0.025), ("4H", 0.015), ("1H", 0.0075), ("15M", 0.004), ("5M", 0.002)]
```

Плюсы: просто, 1 строка.
Минусы: синтетическое раздувание (ты сам критиковал: "НЕ используем синтетическое раздувание — это фейковые зоны").

### C) Sticking detection для 15m (в _fill_missing_tf_zones)

В `eddc8dc` уже есть sticking detection, но он проверяет parent=child. На XAUT `15m lower (4007.45) = 1h lower (4007.45)` — должно сработать, но не сработало. Проверь почему — возможно tolerance 0.5% слишком жёсткая, или 15m не проходит проверку после `del`.

### D) Структурное окно (рекомендуемый, но сложный)

Изменить ZigZag benchmark: вместо `limit=200` брать `limit=40-50` и показать 2 последних структурных движения. В промпте добавить: "Зона = range от последнего слома структуры, не микроканал последних свечей".

Плюсы: фиксит корневую причину — LLM будет видеть реальные структуры.
Минусы: требует изменений в ZigZag + промпт + график.
