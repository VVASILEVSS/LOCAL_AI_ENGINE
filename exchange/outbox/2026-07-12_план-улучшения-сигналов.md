## P0 — Быстрые фиксы метрик (критично)

### P0-1: RSI метод — SMA → Wilder EMA

**Файл:** `core/auto_chart.py:476-479`

```python
# ТЕПЕРЬ (SMA — расходится с TradingView):
delta = df['close'].diff()
gain = delta.where(delta > 0, 0).rolling(14).mean()
loss = (-delta.where(delta < 0, 0)).rolling(14).mean()
```

```python
# НАДО (Wilder EMA — стандарт TradingView):
delta = df['close'].diff()
gain = delta.where(delta > 0, 0).ewm(alpha=1/14, min_periods=14).mean()
loss = (-delta.where(delta < 0, 0)).ewm(alpha=1/14, min_periods=14).mean()
```

**Влияние:** RSI будет совпадать с TradingView. Расхождение 68.3 vs 52.5 исчезнет.

### P0-2: Volume ratio — последняя свеча → среднее последних 3

**Файл:** `core/auto_chart.py:493-494`

```python
# ТЕПЕРЬ (одна свеча — сильно скачет):
vol_ma = df['volume'].rolling(20).mean().iloc[-1]
vol_ratio = df['volume'].iloc[-1] / max(vol_ma, 1e-6)
```

```python
# НАДО (среднее 3 свечей — стабильнее):
vol_ma = df['volume'].rolling(20).mean().iloc[-1]
vol_recent = df['volume'].tail(3).mean()
vol_ratio = vol_recent / max(vol_ma, 1e-6)
```

**Влияние:** vol_ratio перестанет скакать от 0.36 до 1.75 на одной свече.

## P1 — Контекст для LLM (высокий приоритет)

### P1-1: Подключить liquidity_heatmap в промпт

**Проблема:** `liquidity_heatmap.py` (16 KB, 20 функций) существует, но его результаты **не передаются в LLM**. PRO_TA_USER_PROMPT не имеет поля `{liquidity_context}`.

**Решение:**
1. В `analyze_multi_images()` — добавить `liquidity_str = build_liquidity_context_text(prev_analysis.get("liquidity_heatmap"))`
2. В `PRO_TA_USER_PROMPT` — добавить секцию `Liquidity heatmap:\n{liquidity_context}`
3. В `scheduler.py`/`handlers.py` — вызывать `build_liquidity_heatmap(df)` и передавать в `prev_analysis`

**Влияние:** LLM увидит зоны ликвидности, ближайшие уровни, dominant_bias — вместо гадания по графику.

### P1-2: Подключить state_tracker (история зон)

**Проблема:** `state_tracker.py` (20 KB, 15 функций) существует, но `state_context = "{}"` — пусто в прогнозе.

**Решение:**
1. В `scheduler.py` — после получения `tf_zones` вызывать `update_and_save_state(symbol, tf, current_state)`
2. Передавать `state_diff = compare_state(previous, current)` в `prev_analysis["state_context"]`

**Влияние:** LLM увидит какие уровни сломаны, где retest, где false_breakout — из реальной истории, а не из одной свечи.

### P1-3: Open Interest в контексте

**Проблема:** `binance_metrics.py` собирает OI, но в прогнозе `Open Interest: N/A`. ccxt возвращает `openInterestAmount`, но код не парсит правильно.

**Решение:** в `binance_metrics.py` — добавить парсинг:
```python
oi_amount = oi.get('openInterestAmount') or oi.get('baseVolume')
oi_value = oi_amount * current_price if oi_amount else None
```

**Влияние:** LLM увидит реальный OI в BTC и $ — индикатор позиционирования рынка.

### P1-4: Multi-timeframe (добавить 1H + 15m)

**Проблема:** Прогноз только на 4H + 1D. LLM пишет в `missing_data`: "Нет данных по 1H и 15m". Для точных входов нужен младший ТФ.

**Решение:** В `scheduler.py`/pipeline — добавить 1H и 15m графики:
```python
timeframes = [("15m", 200), ("1h", 200), ("4h", 200), ("1d", 200)]
```

**Влияние:** LLM сможет определить точку входа на 15m/1h после подтверждения на 4H/1D.

## P2 — Качество LLM-ответа (средний приоритет)

### P2-1: Temperature = 0.2 (не 0.7)

**Проблема:** `ollama_service.py` не передаёт temperature явно → дефолт 0.7–1.0 → LLM "творит", нестабильные ответы.

**Решение:** В `ollama_service.py` → `generate()`:
```python
payload = {
    "model": model_name,
    "messages": messages,
    "temperature": 0.2,  # детерминированность
    "max_tokens": 4096,
}
```

**Влияние:** Прогноз стабилен при повторных прогонах (тот же вход → тот же выход).

### P2-2: Few-shot примеры в системном промпте

**Проблема:** LLM получает схему JSON, но без примеров "хорошего" ответа.

**Решение:** Добавить 1–2 примера в `PRO_TA_SYSTEM_PROMPT`:
```
ПРИМЕР (коррекция, нет сигнала):
{"price": 63895, "signal_status": "no_signal", "confidence": "low", ...}

ПРИМЕР (пробой с объёмом, LONG):
{"price": 65000, "signal_status": "aggressive_breakout", "entry_conditions": {...}, ...}
```

**Влияние:** LLM будет точнее следовать схеме и логике.

### P2-3: Self-consistency (3 прогона → голосование)

**Проблема:** Один прогон LLM → один ответ. Модель может ошибиться.

**Решение:** В `analyze_multi_images()` — запустить 3 прогона с temperature=0.3, выбрать мажоритарный signal_status:
```python
results = await asyncio.gather(*[llm_generate(...) for _ in range(3)])
signals = [parse_llm_json(r)["signal_status"] for r in results]
final_signal = max(set(signals), key=signals.count)  # голосование
```

**Влияние:** Уберёт случайные галлюцинации, сигнал стабилен.

### P2-4: Валидация логики ответа

**Проблема:** `parse_llm_json` парсит JSON, но не проверяет логику (SL > entry для LONG, TP < SL, и т.д.).

**Решение:** Добавить `validate_signal(data)` в `enforce_risk_rules`:
```python
if signal_status in ("aggressive_breakout", "retest") and entry_direction == "long":
    if sl and sl >= entry_price:
        data["error"] = True
        data["error_message"] = "SL должен быть ниже entry для LONG"
```

**Влияние:** Гарантия что SL/TP логичны, некорректные ответы помечаются.

## P3 — Архитектурные (long-term)

### P3-1: Backtest pipeline (историческая точность)

**Проблема:** `backtest = "Статистика ещё формируется"` — пусто. LLM не знает насколько его прошлые прогнозы были точны.

**Решение:**
1. После каждого прогноза — сохранять `{symbol, tf, signal, price, timestamp}` в БД
2. Через N часов — проверять: цена пошла по прогнозу?
3. Передавать в LLM: "Историческая точность: 67% сигналов верны (last 30)"

**Влияние:** LLM будет калибровать confidence на основе реальной точности.

### P3-2: ZigZag контекст

**Проблема:** `zigzag_context = "{}"` — пусто. Нет волновой разметки.

**Решение:** Реализовать ZigZag индикатор (pivot detection) в `auto_chart.py`:
```python
def calculate_zigzag(df, depth=5, deviation=5):
    pivots = find_pivots(df['high'], df['low'], depth)
    return {"waves": [...], "last_pivot": {...}}
```

**Влияние:** LLM увидит волновую структуру — усилит ABC анализ.

### P3-3: Мульти-символьный confluence

**Проблема:** BTC анализируется изолированно. Но ETH, SOL, DXY влияют на BTC.

**Решение:** Передавать в контекст корреляцию:
```
BTC dominance: 52%, ETH/BTC: -1.2%, DXY: +0.3%
```

**Влияние:** LLM учтёт контекст всего рынка, не только BTC.

### P3-4: A/B тест промптов

**Решение:** Держать 2 версии промпта (A=current, B=improved), запускать оба, сравнивать точность через backtest pipeline (P3-1).

## Резюме

**Корневая причина расхождений** — не LLM, а входные данные:
1. RSI считается по SMA вместо Wilder EMA → расходится с TradingView
2. Volume ratio берёт 1 свечу вместо среднего → скачет
3. Liquidity heatmap (16 KB кода) **не передаётся в LLM**
4. State tracker (20 KB кода) **не передаёт историю**
5. Open Interest = N/A (парсинг баг)
6. Нет 1H/15m для точных входов
7. Temperature не задана → нестабильность
8. Нет backtest → LLM не калибруется

**Минимальный набор для качества:**
- P0 (RSI + volume + ATR) — 6 строк кода, убирает расхождения с TradingView
- P1 (liquidity + state + OI + multi-TF) — ~70 строк, даёт LLM полный контекст
- P2 (temperature + few-shot) — стабилизирует ответы

Всё остальное (backtest, self-consistency, ZigZag) — следующий этап.
