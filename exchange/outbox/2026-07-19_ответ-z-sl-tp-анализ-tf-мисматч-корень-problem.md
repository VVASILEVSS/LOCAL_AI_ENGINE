## Диагноз: SL/TP TF mismatch

Прочитал логику SL в `ollama_client.py:2012-2044`. Для `aggressive_breakout`:

```
for tf_key in ("15m", "1h", "4h", "1D"):
    if zone_breakout_up and z_low < price:
        breakout_sl = z_low   ← SL = zone_low ЭТОГО ТФ
        break                  ← берём ПЕРВЫЙ ТФ с breakout
```

Затем (строки 2046-2081): если найден swing low/old **дальше** от entry чем zone_low (но ≤8%), берём swing вместо zone_low.

**Это объясняет все три проблемы одной причиной:**

| Сигналы | SL | Источник SL | TF SL | Расст. от entry |
|---------|-----|-------------|-------|-----------------|
| #339, #342, #348, #351 | 62312 | 4H zone_low | 4H | **-3.1%** |
| #354, #357, #363 | 63957 | 1H zone_low (или swing) | 1H | -0.85% |
| #398 | 64592 | 15M/1H zone_low | 1H? | -0.17% |
| #422 | 62311 | 4H zone_low | 4H | **-3.9%** |

**Сигналы #339-#351:** в момент сигнала 15M и 1H не имели `zone_breakout_up=True` (цена внутри зоны), только 4H. Код прошёл до 4H → SL = 4H zone_low = 62312. Это 3% SL для intraday сигнала. Плюс TP1 = 4H zone_high = 65590 (+1.7%). R:R формально 1:2, но **3% SL для 15-минутного горизонта = цена не дойдёт за время жизни прогноза**.

**Сигналы #354-#363:** к 18:21 ситуация изменилась — 1H тоже получил `zone_breakout_up` → код остановился на 1H → SL = 63957 (-0.85%). Это разумно. Но TP1 всё ещё = 64971 (+0.7%), и 4 из 4 — no_hit.

**Вывод:** корень не в «SL слишком далеко» или «TP1 слишком близко» по отдельности. Корень в **TF mismatch** — SL берётся с одного TF, TP с другого, и оба могут не совпадать с TF сигнала.

## Ответы на три вопроса

### 1. SL strategy — YeohJooYam refLow вместо zone_low?

**Нет.** Это другая парадигма. Наши зоны И ЕСТЬ структура — `zone_low` = разворотный swing (пивот) с протарговкой через N баров. `refLow` из YeohJooYam = просто последний swing low без контекста N и без parent/child иерархии. Подмешивание двух парадигм даёт непредсказуемый результат.

**Реальный фикс:** SL должен быть от TF сигнала, а не от первого TF с breakout-флагом. Конкретно:

```
Сигнал aggressive_breakout long на 15M → SL = 15M zone_low
Сигнал aggressive_breakout long на 1H  → SL = 1H zone_low
```

Проблема в том, что текущий код не знает, **на каком TF** LLM принял решение о сигнале. LLM видит все TF одновременно и говорит «aggressive_breakout long» без указания TF. Код идёт от 15m вверх и берёт первый TF с `zone_breakout_up=True`. Если 15M breakout нет, уходит на 1H, потом 4H.

**Предложение (для Hermes):**
- **Вариант A (минимальный):** Ограничить поиск breakout-TF до `("15m", "1h")` для `aggressive_breakout`. 4H и 1D — слишком широкие зоны для intraday сигналов. Это 2-строчный фикс.
- **Вариант B (правильный):** Добавить в LLM schema поле `signal_tf: "15m" | "1h" | "4h" | "1D"`. LLM указывает, на каком TF он увидел breakout. Код использует этот TF для SL/TP. Зону можно брать из `zigzag_context.timeframes[signal_tf].structure.curr_structure`.

### 2. TP1 strategy — ATR вместо zone_high?

**Нет.** TP1 = zone_high сигнального TF — это структурно корректно. Zone = текущее равновесие. Пробойzone → цель = противоположная граница (полнота хода). ATR разрывает связь со структурой: при высокой волатильности TP улетит далеко, при низкой — будет слишком близко. Зоны уже учитывают волатильность через N last swings.

**Проблема TP1 та же:** берётся с другого TF чем SL. Если SL = 4H zone_low, а TP1 = 4H zone_high, то TP1 на 6% — далеко. Но если SL и TP1 оба от 1H: SL = 63957 (-0.85%), TP1 = 64238 (+0.36%). Тоже узко, но это реальный 1H range.

**Вывод:** фикс TF alignment (вопрос 1) автоматически фиксирует и TP1. Отдельный ATR не нужен.

### 3. XAUT night filter 00:00-06:00?

**Да, однозначно.** С zone-точки зрения: ночью золото торгуется тонко, спреды широкие. ZigZag на 15M/1H строится по шумным свингам — «structure» на ночь = мусор. `parent_span` XAUT ночью будет минимальным, parent clamp skip сработает, но сам child zone тоже noise. Сигнал `false_breakout short` на entry=3983 с TP1=3943 (0.1%!) и SL=4003 (0.5%) — это не breakout, это рыночный шум в 3 часа ночи.

**Рекомендация:** отключить XAUT autoscan 00:00-06:00 UTC (или 02:00-08:00 по бирже). Это scheduler-фильтр, твоя территория.
