## Задача

Нет механизма сравнения двух версий промпта. P3-1 (Super Z) дал `signal_log` таблицу — теперь можно измерять accuracy. P3-4 добавляет переключение A/B и разбивку статистики.

## Что сделано

### 1. config.py — PROMPT_VARIANT env var

```python
PROMPT_VARIANT = os.getenv("PROMPT_VARIANT", "A").upper().strip()
if PROMPT_VARIANT not in ("A", "B"):
    PROMPT_VARIANT = "A"
```

Переключение через `.env`: `PROMPT_VARIANT=B` или env var.

### 2. ollama_client.py — Variant B промпт + выбор

**Variant A** (current, 6402 chars): strict fact-based + 2 few-shot примера (no_signal + aggressive_breakout).

**Variant B** (new, 1392 chars): rules-based, без few-shot, с явным приоритетом signal_status:

```
ПРИОРИТЕТ SIGNAL_STATUS (выбери ровно один):
1. aggressive_breakout — пробой границы зоны с объёмом >1.5x
2. retest — цена вернулась к пробитой зоне, отскок с объёмом
3. false_breakout — выход за границу + возврат внутрь + объём
4. reversal — смена направления после тренда
5. accumulation — цена в диапазоне, объём <1.0x
6. no_signal — цена между уровнями, пробоя нет
```

**Гипотеза B**: убирая few-shot (экономия ~5000 токенов), LLM получает меньше "якорей" и может лучше адаптироваться к контексту. Явный приоритет signal_status помогает избежать противоречий (false_breakout + accumulation одновременно).

**Экономия:** 6402 → 1392 chars (78%).

Функция `_get_system_prompt()` выбирает промпт:
```python
def _get_system_prompt() -> str:
    if PROMPT_VARIANT == "B":
        return PRO_TA_SYSTEM_PROMPT_B
    return PRO_TA_SYSTEM_PROMPT
```

`analyze_multi_images` использует `_get_system_prompt()` вместо хардкода.

### 3. backtest.py — prompt_variant колонка + A/B статистика

**CREATE TABLE:** добавлено `prompt_variant TEXT DEFAULT 'A'`.

**Миграция existing DB:** `ALTER TABLE signal_log ADD COLUMN prompt_variant TEXT DEFAULT 'A'` (try/except — колонка может уже быть).

**save_signal_log:** новый параметр `prompt_variant: str = "A"`, сохраняется в БД.

**get_backtest_context:** добавлен A/B блок:
```
A/B variants: VA: 1/1 (100%) | VB: 0/1 (0%)
```

SQL-запрос разбивает статистику по variant:
```sql
SELECT prompt_variant, COUNT(*) as cnt,
       SUM(CASE WHEN outcome IN ('tp1_hit','tp2_hit','tp3_hit') THEN 1 ELSE 0 END) as wins
FROM signal_log
WHERE symbol = ? AND checked_at IS NOT NULL
GROUP BY prompt_variant
```

### 4. scheduler.py — передача variant

```python
from core.config import MY_CHAT_ID, PROMPT_VARIANT
...
save_signal_log(parsed, symbol_id, timeframes, prompt_variant=PROMPT_VARIANT)
```

## Как использовать

1. Запустить с variant A (default): `python main.py`
2. Через неделю переключить на B: добавить `PROMPT_VARIANT=B` в `.env`
3. Через неделю сравнить accuracy через `get_backtest_context` — строка `A/B variants: VA: X% | VB: Y%`
4. Или SQL:
   ```sql
   SELECT prompt_variant,
          COUNT(*) as total,
          SUM(CASE WHEN outcome IN ('tp1_hit','tp2_hit','tp3_hit') THEN 1 ELSE 0 END) as wins
   FROM signal_log
   WHERE checked_at IS NOT NULL
   GROUP BY prompt_variant;
   ```
