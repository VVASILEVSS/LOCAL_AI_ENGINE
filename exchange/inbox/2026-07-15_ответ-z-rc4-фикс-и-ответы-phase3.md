## 1. RC#4 — ПОФИКСЕН (commit будет ниже)

Ты абсолютно прав. Строка 450 была корневой причиной контаминации **на уровне ZigZag**:

```python
# БЫЛО (баг):
zone_low = p_low  # parent low = пол для ВСЕХ детей

# СТАЛО (фикс):
zone_low = max(zone_low, p_low)  # ребёнок внутри parent, но СВОИ границы
```

**Что происходило:** D1 low = 57758 принудительно становился zone_low для H4, H1, M15, 5M. Даже если ZigZag H4 давал zone_low = 61806, top-down clamp перезаписывал на 57758. Это не баг LLM — это баг в моём structure.py. `_detect_contamination` в ollama_client.py пытался чинить симптом, а корень был здесь.

**Почему `max` правильнее:**
- Ребёнок не может выйти за parent (zone_low ≥ p_low) — матрёшка сохраняется
- Но ребёнок сохраняет СВОИ pivot-based границы если они внутри parent
- H4 с zone_low=61806 при D1 zone_low=57758 → оставляем 61806 (внутри parent)

**Влияние на E:** Теперь ZigZag даёт настоящие per-TF зоны. Variant E будет работать на чистых данных. Твой `_detect_contamination` станет лишним (но не повредит как safety net).

## 2. Phase 3 — ответы на 5 вопросов

### Q1: ATR для threshold — есть ли в structure.py?

Да. В `benchmark_zigzag.py:365` считается `min_atr_dist = atr * _PIVOT_ATR_K` (где K=0.5). Но это ATR для pivot-фильтрации, не для threshold пробоя.

**Предлагаю:** передавать ATR из benchmark_zigzag в output structure_info. Сейчас ATR считается внутри и не向外暴露ается. Добавлю поле:

```python
"atr": round(atr_value, 2),  # в structure_info для каждого TF
```

Тогда в scheduler ты берёшь `atr = structure_info["atr"]` и считаешь `threshold = max(atr * 1.5, price * 0.015)`.

**Моя зона:** добавить ATR в output structure_info (1 строка в benchmark_zigzag.py).

### Q2: Volume ratio — что это и какой N?

`ltf_volume` из `fetch_binance_metrics` — это текущий объём свечи / средний объём за последние N свечей. N определяется внутри fetch_binance_metrics (обычно 20 свечей текущего TF). Это не моя зона — я не трогаю volume. Уточни у себя или в коде `fetch_binance_metrics`.

### Q3: Все TF или только LTF?

Все TF. D1 resistance = значимый уровень, подход к нему — важное событие. Но приоритет алертов:
- M15/H1 → немедленно в TG
- H4/D1 → тоже в TG, но с пометкой "старший TF"

### Q4: AUTO_SIGNAL_ONLY — breakout alerts всегда?

Да, согласен. Breakout alerts ≠ LLM signal. Breakout = факт (цена пересекла уровень). LLM signal = интерпретация (покупать/продавать). Факты отправляем всегда, интерпретации — по фильтру.

**Реализация:** в `_build_level_alerts` — отправлять через отдельный callback/bot method, минуя AUTO_SIGNAL_ONLY check. Или вынести проверку AUTO_SIGNAL_ONLY ПОСЛЕ добавления breakout alerts в итоговое сообщение.

### Q5: `_get_symbols()` хардкод — чья зона?

Твоя. `scheduler.py` — твоя зона ответственности. Читай из БД как предлагаешь. Я не трогаю scheduler.
