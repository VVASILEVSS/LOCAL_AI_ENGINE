# Обновлённое техническое задание: A/D слой для LOCAL_AI_ENGINE

## 1. Цель
Собрать A/D как **интерпретируемый объёмный контекст**, который:
- определяет bullish / bearish / neutral volume bias;
- показывает volume confirmation;
- выявляет divergence;
- оценивает силу/слабость движения;
- помогает отличать накопление и распределение;
- усиливает или ослабляет сигналы Liquidity Magnet;
- работает на нескольких таймфреймах;
- даёт понятный контекст для LLM, Telegram и визуальной диагностики.

## 3. Скрипты, задействованные в A/D контуре

Ниже — актуальный список ролей.

### 3.1 `core/volume_filters.py`
**Назначение:** основной Python-слой A/D.  
**Должен отвечать за:**
- A/D bias;
- CMF / flow-context;
- volume confirmation;
- divergence detection;
- strength score;
- TF-aware summary;
- интерпретацию объёма относительно цены.

**Связан с:**
- `core/auto_chart.py`
- `core/ollama_client.py`
- `core/wyckoff.py` или `core/market_phase.py`
- `core/liquidity_heatmap.py`

---

### 3.2 `core/auto_chart.py`
**Назначение:** сбор рыночного контекста и графиков.  
**Сейчас отвечает за:**
- получение OHLCV;
- расчёт базовых метрик;
- формирование chart context;
- передачу контекста в LLM;
- использование `OhlcvDataProvider`.

**Нужно доработать для A/D:**
- добавить вызов `analyze_volume_context(...)`;
- передавать A/D summary в `prev_ctx`;
- не ломать Telegram/LM Studio pipeline.

---

### 3.3 `core/ollama_client.py`
**Назначение:** запрос к LLM и сбор JSON-ответа.  
**Сейчас отвечает за:**
- prompt;
- парсинг JSON;
- нормализацию риска;
- форматирование ответа.

**Нужно доработать для A/D:**
- добавить A/D поля в prompt;
- использовать A/D как часть reasoning;
- учитывать divergence / volume confirmation в итоговой интерпретации.

---

### 3.4 `core/state_tracker.py`
**Назначение:** отслеживание изменений состояния между анализами.  
**Связан с:**
- зонами;
- state diff;
- retrace / retest / false breakout / rebuild.

**Нужно для A/D:**
- учитывать, подтверждается ли переход состояния объёмом;
- хранить A/D state snapshot при необходимости.

---

### 3.5 `core/wyckoff.py` или `core/market_phase.py`
**Назначение:** фазовая классификация рынка.  
**Нужно для A/D:**
- связать bias с фазой;
- использовать volume confirmation в accumulation / distribution;
- различать absorption / exhaustion / confirmation.

---

### 3.6 `core/liquidity_heatmap.py`
**Назначение:** ликвидностный слой.  
**Нужно для A/D:**
- сравнивать divergence и volume confirmation с зонами ликвидности;
- усиливать/ослаблять сценарии;
- использовать как контекст для ложных пробоев.

---

### 3.7 `tests/AD/diag_pivots.py`
**Назначение:** диагностика pivot/divergence pipeline.  
**Нужно для A/D:**
- проверка pivot-логики;

---

## 5. Как это встраивается в LOCAL_AI_ENGINE

## Поток данных
1. `core/data_provider.py`
   - обновляет OHLCV CSV в `data/ohlcv/current/`

2. `core/auto_chart.py`
   - читает CSV;
   - считает базовые метрики;
   - вызывает `core/volume_filters.py`

3. `core/volume_filters.py`
   - считает A/D context;
   - возвращает structured summary

4. `core/ollama_client.py`
   - получает:
     - price context
     - zone context
     - ZigZag context
     - A/D context
     - state context
   - строит ответ модели

5. `core/state_tracker.py`
   - отслеживает изменения по уровням и структуре

6. `core/liquidity_heatmap.py` / `core/wyckoff.py`
   - усиливают интерпретацию режима рынка

## 6. Рекомендуемая архитектура A/D слоя

### Вход
- OHLCV
- volume
- timeframe
- ATR
- moving averages
- optional flow smoothing

### Выход
```json
{
  "bias": "bullish|bearish|neutral",
  "confirmation": "strong|weak|none",
  "divergence": "regular_bullish|regular_bearish|hidden_bullish|hidden_bearish|none",
  "strength": 0.0,
  "volume_ratio": 1.42,
  "cmf": 0.031,
  "flow_z": 1.28,
  "regime": "bull|bear|flat|transition",
  "comment": "краткая интерпретация"
}
```

---

## 7. Критерии готовности

A/D слой готов, если:

- bias совпадает с объёмной картиной;
- divergence определяется стабильно;
- confirmation не противоречит цене;
- сильное движение не помечается как weak без причины;
- Python-контекст соответствует Pine-скрипту;
- LLM получает A/D как отдельный и понятный блок;
- A/D помогает фильтровать Liquidity Magnet.
